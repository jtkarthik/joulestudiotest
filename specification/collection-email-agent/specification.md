## Project-Specific Tasks

### AR Data Retrieval Tool

- [ ] Implement `get_customer_open_items` tool in `app/tools/ar_tools.py`:
  - Accepts `customer_id: str` as input
  - Uses MCP tools (loaded via `get_mcp_tools()`) to query open invoices and overdue amounts from SAP S/4HANA
  - Returns structured data: list of open items with `invoice_id`, `amount`, `currency`, `due_date`, `days_overdue`
  - Sets `top=100` on every paginated tool call to prevent context overflow
  - Does NOT call SAP APIs directly — only through MCP tools

### Payment History Tool

- [ ] Implement `get_payment_history` tool in `app/tools/ar_tools.py`:
  - Accepts `customer_id: str` as input
  - Uses MCP tools to fetch payment history from Payment Advice API (`CE_PAYMENTADVICE_0001`)
  - Returns: list of past payments with `payment_date`, `amount`, `currency`, `invoice_id`, `days_late` (negative = early)
  - Sets `top=100` on every paginated tool call

### Dunning Status Tool

- [ ] Implement `get_dunning_status` tool in `app/tools/ar_tools.py`:
  - Accepts `customer_id: str` as input
  - Uses MCP tools to read dunning records from Contract Accounting Dunning API (`CADUNNING_0001`)
  - Returns: `dunning_level`, `last_dunning_date`, `dunning_blocked` flag

### Payment Behavior Analysis

- [ ] Implement `analyze_payment_behavior(open_items, payment_history, dunning_status) -> dict` in `app/tools/analysis.py`:
  - Pure Python function (no LLM, no MCP calls)
  - Computes: `total_overdue_amount`, `max_days_overdue`, `avg_days_late`, `payment_reliability_score` (0–100), `urgency_level` ("reminder_1" | "reminder_2" | "escalation")
  - Urgency rules:
    - `reminder_1`: first overdue, avg_days_late ≤ 15
    - `reminder_2`: recurring late payments or avg_days_late > 15
    - `escalation`: dunning_level ≥ 2 or max_days_overdue > 60
  - This function is instrumented with M2 milestone logging

### Email Draft Generation (Agent System Prompt + LLM)

- [ ] Write a runtime skill at `app/skills/collection-email-drafting/SKILL.md` containing:
  - Detailed instructions for the LLM on how to compose collection emails
  - Tone guidance per urgency level: professional/friendly for reminder_1, firm for reminder_2, escalation for escalation
  - Template structure: subject line, salutation, invoice table (invoice_id, amount, due_date), overdue summary, payment options / CTA, closing
  - Instruction to always reference specific invoice numbers and exact overdue amounts
  - Instruction to personalize based on payment reliability score (acknowledge good history if score > 70)

- [ ] In `app/agent.py`, update the `@prompt_section` function to load the collection-email-drafting runtime skill and incorporate it into the system prompt

### Agent Flow (agent.py)

- [ ] Wire tools into the agent: `get_customer_open_items`, `get_payment_history`, `get_dunning_status`
  - Load tools via `get_mcp_tools()` — lazy, cached
  - Also include the `analyze_payment_behavior` function as a local tool (registered as `StructuredTool` without MCP)
- [ ] Extract all business logic from `stream()` into `_run_agent(query, context_id)` async helper
- [ ] Instrument `_run_agent` with OpenTelemetry span: `tracer.start_as_current_span("collection-email-agent.run")`
- [ ] Add milestone instrumentation inside `_run_agent`:
  - **M1**: after successful AR data retrieval — `logger.info("M1.achieved: AR data retrieved for customer %s", customer_id)` — on failure: `logger.warning("M1.missed: AR data retrieval failed for customer %s — %s", customer_id, error)`
  - **M2**: after payment behavior analysis — `logger.info("M2.achieved: payment behavior analyzed — urgency=%s for customer %s", urgency_level, customer_id)` — on skip: `logger.warning("M2.missed: payment behavior analysis skipped for customer %s", customer_id)`
  - **M3**: after email draft generated — `logger.info("M3.achieved: email draft generated for customer %s", customer_id)` — on failure: `logger.warning("M3.missed: email draft generation failed for customer %s — %s", customer_id, error)`
  - **M4**: after draft surfaced for review — `logger.info("M4.achieved: draft reviewed by specialist — action=%s for customer %s", action, customer_id)` — on miss: `logger.warning("M4.missed: draft not reviewed — no action taken for customer %s", customer_id)`
  - **M5**: after email dispatched — `logger.info("M5.achieved: collection email sent to customer %s", customer_id)` — on miss: `logger.warning("M5.missed: email dispatch failed or cancelled for customer %s", customer_id)`

### Guardrails in System Prompt

- [ ] Include these guardrails in the agent's system prompt:
  - Never send or simulate sending emails — output is always a draft for human review
  - Never fabricate invoice numbers, amounts, or dates — use only data from tool responses
  - If tool returns no data for a customer, respond that no AR data was found and halt draft generation
  - Set `top=100` on all paginated MCP tool calls
